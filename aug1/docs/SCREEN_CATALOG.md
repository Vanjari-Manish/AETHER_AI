## 1. Authentication Screens

### 1.1 Login Screen
*   **Purpose:** Authenticates registered utility operators, compliance engineers, and administrators to grant secure access to the GPO console.
*   **Target Users:** Grid Operators, Utility Engineers, Analysts, Researchers, Administrators.
*   **Entry Points:** Initial application URL access (default root redirect), logout redirect, session expiration timeout.
*   **Navigation:** 
    *   *From:* Root URL redirection.
    *   *To:* Main Application Dashboard (on success), Signup Screen, Forgot Password Screen.
*   **Major Sections:**
    *   *System Branding Area:* Displays the official GPO brand logomark and Tagline to establish immediate application context.
    *   *Credentials Input Form:* Secure text fields capturing User ID, password entry, and corporate domain code.
    *   *Security Warning Disclaimer:* A legally required banner warning that the system is connected to critical electrical infrastructure and unauthorized access is federally prosecuted under NERC CIP rules.
*   **Components Used:** Forms, Buttons, Inputs, Text Fields, Badges, Cards.
*   **Required APIs:** Authentication API.
*   **Required Permissions:** Guest (no active session).

### 1.2 Signup Screen
*   **Purpose:** Allows newly authorized utility personnel to register their profile in the system database using verified corporate credentials.
*   **Target Users:** Utility Engineers, Compliance Officers, Grid Operators.
*   **Entry Points:** Click redirect from Login Screen.
*   **Navigation:**
    *   *From:* Login Screen.
    *   *To:* Login Screen, Verify Email Screen.
*   **Major Sections:**
    *   *Credentials Setup Form:* Fields for entering full name, corporate email address, employee ID, role classification, password selection, and security token.
    *   *Security Token Verification Field:* A text input requiring the validation of a physical token generated by the utility's internal Active Directory server.
*   **Components Used:** Forms, Buttons, Inputs, Cards.
*   **Required APIs:** Authentication API.
*   **Required Permissions:** Guest (no active session).

### 1.3 Forgot Password Screen
*   **Purpose:** Allows registered users to request a secure password reset link if they lose access to their credentials.
*   **Target Users:** All user groups.
*   **Entry Points:** Click redirect from Login Screen.
*   **Navigation:**
    *   *From:* Login Screen.
    *   *To:* Login Screen, Reset Password Screen (via email link).
*   **Major Sections:**
    *   *Reset Request Form:* Input capturing registered corporate email address and security token codes.
    *   *Confirmation Instructions:* Descriptive text explaining how the user can verify their request and contact the utility helpdesk if the link fails.
*   **Components Used:** Forms, Buttons, Inputs, Cards.
*   **Required APIs:** Authentication API.
*   **Required Permissions:** Guest (no active session).

### 1.4 Reset Password Screen
*   **Purpose:** Allows users to securely establish a new password after verifying their identity via reset link or token.
*   **Target Users:** All user groups.
*   **Entry Points:** Secure email reset token link.
*   **Navigation:**
    *   *From:* Forgot Password email link.
    *   *To:* Login Screen (post reset).
*   **Major Sections:**
    *   *New Password Entry Form:* Inputs capturing the new password, confirmation entries, and original reset token code.
    *   *Password Rules Checklist:* Dynamic checklist validating password complexity requirements (minimum characters, symbols, numbers, and lack of common patterns).
*   **Components Used:** Forms, Buttons, Inputs, Cards.
*   **Required APIs:** Authentication API.
*   **Required Permissions:** Guest (no active session).

### 1.5 Verify Email Screen
*   **Purpose:** Verifies the user's email address by requiring a one-time validation code sent to their registered corporate email.
*   **Target Users:** New users completing registration.
*   **Entry Points:** Redirect after successful submission on the Signup Screen.
*   **Navigation:**
    *   *From:* Signup Screen.
    *   *To:* Login Screen (post verification).
*   **Major Sections:**
    *   *Verification Code Form:* Numeric inputs for the one-time passcode.
    *   *Resend Request Action:* A link triggering a new verification code if the email was not received.
*   **Components Used:** Forms, Buttons, Inputs, Cards.
*   **Required APIs:** Authentication API.
*   **Required Permissions:** Guest (no active session).

## 2. Main Application Screens

### 2.1 Home Screen
*   **Purpose:** Serves as the landing hub for authenticated users, presenting system health metrics, shortcuts to active workflows, and customized widgets based on roles.
*   **Target Users:** All authenticated users.
*   **Entry Points:** Successful login, navigation from header logo.
*   **Navigation:**
    *   *From:* Sidebar navigation, Header brand logo.
    *   *To:* Dashboard, Topology Map, Policy Studio, Simulation Sandbox, User Profile, Settings.
*   **Major Sections:**
    *   *Unified Command Center Hub:* A central directory layout with quick links to main grid modules (Telemetry, Policy Compiler, Digital Twin).
    *   *Global Grid Health Status:* Aggregated stats showing overall active substations, generation reserves, and pending compliance rules.
    *   *User Worklist:* A checklist of pending tasks, active simulation trials, or policies awaiting approval.
*   **Components Used:** Sidebar, Navbar, Cards, KPI Cards, Buttons, Search Bar, Notification Panel.
*   **Required APIs:** Authentication API, Notification API, Scenario API.
*   **Required Permissions:** Authenticated User, Read Only access minimum.

### 2.2 Dashboard Screen
*   **Purpose:** Real-time operational display showing power grid status, active DER integration levels, and active multi-agent negotiations.
*   **Target Users:** Grid Operators, Utility Engineers.
*   **Entry Points:** Sidebar menu link, Home screen redirect.
*   **Navigation:**
    *   *From:* Sidebar, Home Screen, Alarms.
    *   *To:* Digital Twin Map, AI Agents Panel, Timeline, Audit Logs, Settings.
*   **Major Sections:**
    *   *Real-Time System Alarm Strip:* A prominent warning banner at the top showing current critical failures or grid line sags.
    *   *Main Telemetry KPI Panel:* Large metrics displaying system frequency (Hz), active load (MW), and voltage bounds.
    *   *Live Operations Flowchart:* An interactive mini-topology diagram showing current power routing paths.
    *   *AI Agent Mitigation Feed:* A side panel detailing active agent negotiations, consensus progress, and policy approvals.
*   **Components Used:** Sidebar, Navbar, Cards, KPI Cards, Status Indicators, Charts, Tables, Agent Cards, Notification Panel, Buttons.
*   **Required APIs:** Scenario API, Notification API, Policy API, Analytics API.
*   **Required Permissions:** Operator, Engineer, Administrator. Read-Write access required for override buttons.

### 2.3 Digital Twin Screen (Grid Topology Map)
*   **Purpose:** Interactive visual model of the grid network's electrical schematic, displaying node connections, lines, transformers, and breaker states.
*   **Target Users:** Grid Operators, Utility Engineers, Analysts.
*   **Entry Points:** Sidebar menu link, Dashboard click events on specific nodes.
*   **Navigation:**
    *   *From:* Sidebar, Dashboard, Scenario Builder.
    *   *To:* Dashboard, AI Agents, Settings, Scenario Builder, Timeline.
*   **Major Sections:**
    *   *Schematic Topology Canvas:* An interactive single-line network diagram displaying grid buses, generation nodes, line loads, and breaker states.
    *   *Node Property Side-Drawer:* A slide-out panel showing the selected node's voltage, active generation capacity, and temperature logs.
    *   *Layer Configuration Controls:* Filters to toggle visibility of DER types, battery storage modules, and physical perimeters.
*   **Components Used:** Sidebar, Navbar, Drawers, Cards, KPI Cards, Buttons, Status Indicators, Tooltips, Search Bar, Filter Panel.
*   **Required APIs:** Scenario API, Analytics API, Policy API.
*   **Required Permissions:** Operator, Engineer, Administrator. Read-Write required to change breaker states.

### 2.4 Scenario Builder Screen
*   **Purpose:** Allows engineers and analysts to configure grid contingency tests, power sags, or weather models for simulation testing.
*   **Target Users:** Utility Engineers, Researchers, Analysts.
*   **Entry Points:** Sidebar menu link, Digital Twin Map action triggers.
*   **Navigation:**
    *   *From:* Sidebar, Digital Twin Map, Simulation Screen.
    *   *To:* Simulation Screen, Policy Engine Screen, Audit Logs.
*   **Major Sections:**
    *   *Scenario Configuration Panel:* Form controls to input parameters like target substation line failure, DER drop percentage, and starting generation reserves.
    *   *Contingency Schedule Timeline:* A chronological path builder defining the timeline of simulated grid events.
    *   *Active Rules Scope Selector:* A list of policies to enforce during the simulation run.
*   **Components Used:** Sidebar, Navbar, Cards, Forms, Inputs, Dropdowns, Buttons, Tables, Timeline, Checkboxes.
*   **Required APIs:** Scenario API, Policy API.
*   **Required Permissions:** Engineer, Administrator. Read-Write access.

### 2.5 Simulation Screen
*   **Purpose:** Executes configured scenarios in the simulator, presenting real-time graphs and multi-agent mitigation logs.
*   **Target Users:** Utility Engineers, Researchers, Analysts.
*   **Entry Points:** Redirect after clicking run in the Scenario Builder.
*   **Navigation:**
    *   *From:* Scenario Builder, Sidebar.
    *   *To:* Scenario Builder, Policy Engine, Reports Screen, Timeline.
*   **Major Sections:**
    *   *Active Run Control Panel:* Buttons to play, pause, or step through the simulation timeline.
    *   *Simulated Telemetry Graphs:* Real-time updating line charts tracking voltage levels, grid frequency, and restoration speeds.
    *   *Coordinated Agent Log Console:* A live terminal console showing step-by-step agent negotiations and compliance checks.
*   **Components Used:** Sidebar, Navbar, Cards, KPI Cards, Charts, Tables, Timeline, Progress Bars, Status Indicators, Buttons, Modal.
*   **Required APIs:** Simulation API, Notification API, Policy API.
*   **Required Permissions:** Operator, Engineer, Administrator. Read-Write access required to run simulations.

### 2.6 Policy Engine Screen (Policy Studio)
*   **Purpose:** Allows compliance officers and engineers to create, edit, compile, and hot-swap policy files that govern agent actions.
*   **Target Users:** Compliance Officers, Utility Engineers.
*   **Entry Points:** Sidebar menu link, Simulation results overview.
*   **Navigation:**
    *   *From:* Sidebar, Simulation, Audit Logs.
    *   *To:* Scenario Builder, Simulation, Audit Logs, Settings.
*   **Major Sections:**
    *   *Interactive Rules Workspace:* Code text editor displaying the policy file configuration with syntax highlighting and auto-fill helper menus.
    *   *Policy Compiler Output Panel:* Monospaced terminal window showing syntax validation logs, error codes, and compiler status.
    *   *Active Constraint Directory:* Directory tree showing currently deployed, pending, and retired policy files.
*   **Components Used:** Sidebar, Navbar, Cards, Forms, Text Fields, Buttons, Tables, Status Indicators, Progress Bars, Dialogs.
*   **Required APIs:** Policy API, Report API.
*   **Required Permissions:** Compliance Architect, Engineer, Administrator. Read-Write required to compile/deploy new policies.

### 2.7 AI Agents Screen
*   **Purpose:** Displays status, decision parameters, weights, and communication logs of all active edge software agents.
*   **Target Users:** Utility Engineers, Grid Operators, Analysts.
*   **Entry Points:** Sidebar menu link, Dashboard click events on agent cards.
*   **Navigation:**
    *   *From:* Sidebar, Dashboard, Digital Twin Map.
    *   *To:* Digital Twin Map, Dashboard, Timeline, Audit Logs.
*   **Major Sections:**
    *   *Agent Coordination Grid:* Panel cards showing current status, load allocation weights, and communication link health for all active agents.
    *   *Consensus Log Table:* A tabular log showing active agent negotiations and pricing offers during the last grid event.
    *   *Agent Settings Drawer:* Collapsible side drawer to configure negotiation speed limits and local voltage thresholds.
*   **Components Used:** Sidebar, Navbar, Cards, Agent Cards, Tables, Status Indicators, KPI Cards, Drawers, Search Bar, Buttons.
*   **Required APIs:** Scenario API, Analytics API.
*   **Required Permissions:** Operator, Engineer, Administrator. Read-Write access to modify agent parameters.

### 2.8 Analytics Screen
*   **Purpose:** Provides tools for long-term grid performance analysis, trend reporting, and DER efficiency metrics.
*   **Target Users:** Analysts, Researchers, Utility Engineers.
*   **Entry Points:** Sidebar menu link, Reports view.
*   **Navigation:**
    *   *From:* Sidebar, Reports Screen.
    *   *To:* Reports Screen, Audit Logs.
*   **Major Sections:**
    *   *Historical Trend Charts:* Custom line and bar charts tracking grid stability indicators over weeks, months, or years.
    *   *DER Integration Heatmaps:* Multi-dimensional heatmaps showing load sags and feed-in power levels.
    *   *Analysis Filter Controls:* Dropdowns and search boxes to narrow analytical queries by date, substation, or rule classification.
*   **Components Used:** Sidebar, Navbar, Cards, Charts, Tables, Dropdowns, Date Range Inputs, Search Bar, Buttons, Filter Panel.
*   **Required APIs:** Analytics API, Report API.
*   **Required Permissions:** Analyst, Engineer, Administrator. Read-Only access.

### 2.9 Reports Screen
*   **Purpose:** Configures, generates, and exports formal compliance and audit logs (e.g. for NERC compliance audits).
*   **Target Users:** Compliance Officers, Administrators, Analysts.
*   **Entry Points:** Sidebar menu link, Analytics Screen actions.
*   **Navigation:**
    *   *From:* Sidebar, Analytics, Audit Logs.
    *   *To:* Analytics, Audit Logs, Settings.
*   **Major Sections:**
    *   *Report Template Builder:* Form elements to select date ranges, target substations, validation criteria, and export format (CSV/PDF).
    *   *Generated Report Archive:* Table listing previously compiled reports with download, archive, and delete actions.
    *   *Signature Validation Box:* Input to enter cryptographic compliance signatures before report generation.
*   **Components Used:** Sidebar, Navbar, Cards, Forms, Inputs, Dropdowns, Buttons, Tables, Checkboxes, Dialogs.
*   **Required APIs:** Report API, Policy API.
*   **Required Permissions:** Compliance Officer, Administrator. Read-Write access.

### 2.10 Timeline Screen
*   **Purpose:** Real-time chronological overview of all system events, agent decisions, compliance checks, and operator actions.
*   **Target Users:** Grid Operators, Utility Engineers, Compliance Officers.
*   **Entry Points:** Sidebar menu link, Dashboard alerts list.
*   **Navigation:**
    *   *From:* Sidebar, Dashboard, Audit Logs, Alarms.
    *   *To:* Dashboard, Digital Twin Map, Policy Engine, AI Agents, Audit Logs.
*   **Major Sections:**
    *   *Chronological Event Track:* A vertical path timeline listing grid alarms, policy compiled events, agent decisions, and operator overrides in order.
    *   *Event Filter Panel:* Checkboxes to toggle visibility of alarms, user logs, agent events, and database actions.
    *   *Telemetry Detail Box:* Displays telemetry logs for the selected event in JetBrains Mono.
*   **Components Used:** Sidebar, Navbar, Cards, Timeline, Tables, Checkboxes, Buttons, Search Bar, Drawers.
*   **Required APIs:** Notification API, Scenario API, Analytics API.
*   **Required Permissions:** Operator, Engineer, Compliance Officer, Administrator. Read-Only access.

### 2.11 Audit Logs Screen

*   **Required Permissions:** Compliance Officer, Administrator, System Administrator. Read-Only access.

## 3. Logically Added Enterprise Screens

To meet the requirements of a mission-critical utility control room and NERC compliance guidelines, the following screens are included in the catalog:

### 3.1 GIS Grid Map Screen (Geographic Visualizer)
*   **Rationale:** Grid operators require a Geographic Information System (GIS) overlay map to locate physical substations, track path structures, and view weather overlays (storm paths, fire zones) alongside schematic data.
*   **Purpose:** Presents a geographic layout of grid infrastructure overlaid with real-time operational status and weather data.
*   **Target Users:** Grid Operators, Utility Engineers, Analysts.
*   **Entry Points:** Navigation link inside the Digital Twin Screen, Sidebar menu link.
*   **Navigation:**
    *   *From:* Digital Twin Screen, Sidebar.
    *   *To:* Digital Twin Screen, Dashboard, Notifications.
*   **Major Sections:**
    *   *Geographic Map Canvas:* A full-screen geographical map displaying substation markers, transmission lines, and real-time weather layers.
    *   *Substation Overlay Cards:* Floating telemetry boxes showing selected station stats (KV load, active alarms).
    *   *Environmental Controls:* Filter options to toggle overlays for wind speeds, temperature heatmaps, and lightning strike areas.
*   **Components Used:** Sidebar, Navbar, Cards, Buttons, Search Bar, Filter Panel, Status Indicators, Tooltips, Drawers.
*   **Required APIs:** Scenario API, Analytics API.
*   **Required Permissions:** Operator, Engineer, Administrator. Read-Only access.

### 3.2 Device Manager Screen (Protocol & SCADA Monitor)
*   **Rationale:** Utility engineers require a dedicated dashboard to monitor real-time hardware connections, verify SCADA link health, and troubleshoot packet latency.
*   **Purpose:** Monitors and configures device connections, DNP3 controllers, and active modbus ports.
*   **Target Users:** Utility Engineers.
*   **Entry Points:** Settings Screen, Sidebar menu link.
*   **Navigation:**
    *   *From:* Settings Screen, Sidebar.
    *   *To:* Settings Screen, Dashboard, Audit Logs.
*   **Major Sections:**
    *   *Connection Status Registry:* A dense table showing all grid RTUs, PLCs, and smart meters with active ping, packet drop rate, and connection status.
    *   *SCADA Polling Controls:* Inputs to modify command validation parameters and telemetry polling rates.
    *   *Protocol Monitor Console:* Live console output displaying raw hex packet traces for active communication channels.
*   **Components Used:** Sidebar, Navbar, Cards, Tables, Status Indicators, Progress Bars, Buttons, Search Bar, Tabs, Drawers.
*   **Required APIs:** Scenario API.
*   **Required Permissions:** Engineer, System Administrator. Read-Write access.

### 3.3 Emergency Override Gate Screen
*   **Rationale:** During severe grid events, operators require a single-purpose interface to bypass the AI agent layers and policy engines entirely, issuing hard manual commands to hardware relays under NERC emergency rules.
*   **Purpose:** Secure control gate to bypass the policy engine and directly trigger breaker states during emergencies.
*   **Target Users:** Grid Operators.
*   **Entry Points:** Global alarm banner links, physical security key switch activation.
*   **Navigation:**
    *   *From:* Dashboard, Global Alerts.
    *   *To:* Dashboard, Digital Twin Map.
*   **Major Sections:**
    *   *Override Authorization Panel:* Inputs requiring verification from two separate operators to confirm the override.
    *   *Manual Relay Controller:* Clean list of physical line breakers with direct manual toggle switches.
    *   *NERC Emergency Logger:* Text field requiring input of the NERC emergency incident ID and a description of the manual bypass.
*   **Components Used:** Navbar, Cards, Forms, Inputs, Buttons, Switches, Status Indicators, Dialogs.
*   **Required APIs:** Scenario API, Authentication API, Report API.
*   **Required Permissions:** Operator, Administrator. Dual-authorization required (Operator + Administrator signatures). Full Read-Write access.
